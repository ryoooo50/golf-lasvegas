# 要件定義書

## はじめに

Golf Las Vegas アプリは、ゴルフの賭けゲーム「ラスベガス」の複雑な計算（バーディーフリップ・プッシュ倍率・キャリーオーバー）を自動化するスマートフォンアプリである。現行実装にはビジネスロジックのバグ、UI の未実装機能、データ消失リスク、および保守性の低いコード構造が存在する。本ブラッシュアップにより、仕様通りの正確な計算・直感的なUX・データ永続化・高い保守性を実現する。

---

## 要件

---

### 要件1: キャリーオーバー倍率の正確な計算

**目的:** ゴルファーとして、キャリーオーバーの倍率が仕様通りに増加してほしい。計算ミスによる金銭トラブルを防ぐため。

#### 受け入れ基準

1. WHEN ホールが引き分けになった場合 AND 現在のキャリーオーバー倍率が1のとき THEN アプリは次のホールの倍率を2に設定する
2. WHEN ホールが引き分けになった場合 AND 現在のキャリーオーバー倍率が2以上のとき THEN アプリは次のホールの倍率を（現在の倍率 + 2）に設定する（例: x2→x4→x6→x8）
3. WHEN ホールで勝敗が決まった場合 THEN アプリは次のホールの倍率を1にリセットする
4. IF キャリーオーバーが3連続で発生した場合 THEN 倍率はx2→x4→x6と正しく増加する

---

### 要件2: プッシュ倍率の正確な計算

**目的:** ゴルファーとして、プッシュ宣言の人数に応じた倍率が正しく計算されてほしい。

#### 受け入れ基準

1. WHEN 誰もプッシュを宣言しないとき THEN PushMult = 0 とする（倍率への加算なし）
2. WHEN n人がプッシュを宣言したとき THEN PushMult = n × 2 とする（1人→2, 2人→4, 3人→6, 4人→8）
3. WHEN プッシュ・キャリーオーバー・イーグルを組み合わせるとき THEN 最終倍率は以下の加算式で計算する（乗算ではなく）:
   ```
   multiplier = max(1, PushMult + COMult + EagleMult)
   PushMult  = pushCount × 2      // プッシュ宣言人数 × 2
   COMult    = COLevel × 2        // キャリーオーバー発生回数 × 2（0, 2, 4, 6...）
   EagleMult = eagleCount × 2     // イーグル以下（ホールインワン含む）の人数 × 2
   ```
4. WHEN プッシュもキャリーオーバーもイーグルも存在しないとき THEN アプリは倍率を1倍として計算する（max(1, 0) = 1）
5. WHEN 個人スコアが9打を超えるとき THEN アプリは計算上のスコアを9にキャップする

---

### 要件3: バーディーの役割分離（フリップのみ・倍率影響なし）

**目的:** ゴルファーとして、バーディーはスコアのフリップのみに影響し、倍率計算には影響しないという仕様通りの動作を期待する。

#### 受け入れ基準

1. WHEN チームAがバーディーを取ったとき THEN アプリはチームBのスコアを反転させる（min桁とmax桁を入れ替え。例: [4,6]→46→64）
2. WHEN チームBがバーディーを取ったとき THEN アプリはチームAのスコアを反転させる
3. WHEN 両チームがバーディーを取ったとき THEN アプリは各チームの元スコアを基準に同時フリップする（連鎖適用しない）
4. WHEN バーディーが発生したとき THEN アプリはバーディーを倍率計算（EagleMult以外）に含めない
5. WHEN バーディーなし・プッシュなし・キャリーオーバーなしのとき THEN アプリは倍率を1倍として計算する

---

### 要件3a: イーグル・ホールインワンのルール（フリップ＋倍率加算）

**目的:** ゴルファーとして、イーグル以下の好スコアが出た場合にフリップに加えて倍率ボーナスが正しく計算されてほしい。

#### 受け入れ基準

1. WHEN プレイヤーがイーグル（パー−2）またはそれ以下（アルバトロス・ホールインワン等）を取ったとき THEN アプリは相手チームのスコアをフリップさせ、かつ EagleMult += 2 を加算する
2. WHEN 同一ホールで複数プレイヤーがイーグル以下を取ったとき THEN アプリはプレイヤーごとに EagleMult += 2 を加算する（例: 2人イーグル → EagleMult = 4）
3. WHEN イーグルとプッシュとキャリーオーバーが同時に発生したとき THEN アプリは `multiplier = max(1, PushMult + COMult + EagleMult)` の加算式で計算する
4. WHEN バーディー（パー−1）が発生したとき THEN アプリは EagleMult を加算しない（フリップのみ）
5. WHEN イーグルが発生したとき THEN アプリはイーグルプレイヤーのチームではなく、相手チームのスコアをフリップする

---

### 要件4: パー入力・バーディー/イーグル判定UI

**目的:** ゴルファーとして、ホールのパーを入力し、バーディー/イーグル判定を自動かつ手動補正できるようにしてほしい。

#### 受け入れ基準

1. WHEN スコア入力画面が表示されているとき THEN アプリは当該ホールのパー入力欄を表示する（デフォルト: 4）
2. WHEN パーが入力されているとき THEN アプリはプレイヤーのスコアと比較し判定を自動設定する:
   - スコア = par − 1 → バーディー（フリップのみ）
   - スコア ≤ par − 2 → イーグル（フリップ + EagleMult += 2）
3. WHEN スコア入力画面が表示されているとき THEN アプリは各プレイヤーのスコア入力欄にバーディー/イーグル判定インジケーターを表示する
4. WHEN 判定インジケーターが手動で変更されたとき THEN アプリは自動判定より手動入力を優先して扱う
5. WHEN バーディー/イーグルが確定したとき THEN アプリは相手チームのスコアが反転することをリアルタイムプレビューに反映する
6. IF コース情報APIは利用しない THEN パーはユーザーが各ホールで手動入力する（オフライン動作を維持するため）

---

### 要件5: リアルタイム計算プレビュー

**目的:** ゴルファーとして、スコア入力中に「現在の入力でどちらが何ポイント勝つか」をリアルタイムで確認できるようにしたい。入力ミスや計算ミスをその場で防ぐため。

#### 受け入れ基準

1. WHILE スコア入力中 THE アプリは現在の入力に基づいた予測結果をリアルタイムで表示する
2. WHEN 4人全員のスコアが入力されたとき THEN アプリは予測される勝利チーム・ポイント差・適用倍率を表示する
3. WHEN スコアが変更されるたびに THEN アプリは予測結果を自動更新する
4. WHEN 引き分けが予測されるとき THEN アプリは「引き分け（キャリーオーバー発生）」と次ホールの倍率を表示する
5. IF スコアが未入力のプレイヤーが存在する場合 THEN アプリはプレビューを非表示または計算不能として表示する

---

### 要件6: 計算結果の詳細内訳表示

**目的:** ゴルファーとして、ホール完了後に「なぜこのポイントになったか」の計算式を確認できるようにしたい。プレイヤー間の透明性を確保するため。

#### 受け入れ基準

1. WHEN ホールのスコアが確定したとき THEN アプリは計算式の内訳を表示する（例: `64 - 45 = 19 × (Push×2 + CO×2 + Eagle×2) = 76pt`）
2. WHEN バーディー/イーグルフリップが発生したとき THEN アプリはフリップ前後のスコアを強調表示する（例: 46 → **64**）
3. WHEN スコアカードでホール履歴を表示するとき THEN アプリは各ホールの勝敗・獲得ポイント・倍率内訳（PushMult・COMult・EagleMult）・バーディー/イーグル発生有無・ホールのチーム編成を表示する
4. IF ホールが引き分けだった場合 THEN アプリは「引き分け - ポイント移動なし」と次ホールキャリーオーバー倍率を表示する
5. WHEN スコアカードを表示するとき THEN アプリは各プレイヤーの累計ポイント推移をデフォルトでは表示しない（詳細展開で閲覧可能とする）
6. WHEN 3人モードのスコアカードを表示するとき THEN アプリはそのホールのソロプレイヤーとポイント×2が適用されたことを明示する

---

### 要件7: データ永続化

**目的:** ゴルファーとして、アプリを閉じても・再起動しても進行中のゲームデータが残ってほしい。途中でアプリが落ちてもデータが消えないようにするため。

#### 受け入れ基準

1. WHEN ホールのスコアが確定するたびに THEN アプリはゲーム状態をAsyncStorageに自動保存する
2. WHEN アプリが再起動されたとき THEN アプリはAsyncStorageから最後のゲーム状態を復元する
3. WHEN 過去のラウンドが保存されたとき THEN アプリは履歴をAsyncStorageに永続化する
4. IF 保存データが破損している場合 THEN アプリは初期状態に安全にフォールバックし、エラーを表示しない
5. WHEN 新しいゲームを開始するとき THEN アプリは進行中のゲームデータをクリアし、保存済みラウンド履歴は保持する

---

### 要件7a: ゲスト利用とログイン任意化

**目的:** ゴルファーとして、アカウント登録やログインをしなくても即座にスコア計算を開始したい。ゴルフ場での利用開始時の摩擦を最小化するため。

#### 受け入れ基準

1. WHEN 未ログインユーザーがアプリを開いたとき THEN アプリはログインを必須にせず、ゲストとしてSTARTできる導線を表示する
2. WHEN ゲストユーザーがプレー中の場合 THEN アプリはスコア入力・リアルタイムプレビュー・ホール確定・スコアカード表示をログインなしで利用できる
3. WHEN ゲストユーザーがラウンド保存・履歴同期・マイページ表示を実行しようとしたとき THEN アプリはログインまたは新規登録を促す
4. WHEN ゲストユーザーがログインしたとき THEN アプリは現在プレー中のゲーム状態を失わず、ラウンド完了後に「保存する」を押した時点でSupabaseにクラウド保存する（ログイン時点での自動同期は行わない）
5. IF ゲストユーザーがログインせずにブラウザデータを削除した場合 THEN アプリはクラウド復元できないことを前提とする

---

### 要件7b: ログインユーザーごとのクラウド履歴保存

**目的:** ログインユーザーとして、過去ラウンド履歴を端末に依存せず保存・参照したい。スマホ変更や別ブラウザ利用時にも履歴を確認できるようにするため。

#### 受け入れ基準

1. WHEN ユーザーが新規登録したとき THEN アプリはSupabase Authにユーザーを作成する
2. WHEN ユーザーがログインしたとき THEN アプリはSupabase Authのセッションを復元し、ログインユーザーとして扱う
3. WHEN ログインユーザーがラウンドを保存したとき THEN アプリはSupabase DBに以下のフィールドを含むラウンド履歴を保存する:
   - `id`, `user_id`, `created_at`, `updated_at`
   - `match_name`（試合名）, `rate`（レート・単位なし）, `player_count`（3 or 4）
   - `player_names`（プレイヤー名JSON）
   - `push_limit`（プッシュ上限）, `birdie_push_recovery`（バーディー復活オプション）
   - `holes`（全ホールのスコア・倍率・チーム編成・バーディー/イーグル情報をJSON）
   - `total_points`（プレイヤーごとの合計ポイントJSON）
3a. WHEN ログインユーザーが保存済みラウンドを後から編集したとき THEN アプリはSupabase DBの該当レコードを更新（UPDATE）できる（スコア修正・未入力ホールの追記を含む）
4. WHEN ログインユーザーがマイページを開いたとき THEN アプリは自分のラウンド履歴のみを表示する
5. WHEN 別ユーザーでログインしたとき THEN アプリは前ユーザーの履歴を表示しない
6. WHERE Supabaseの`rounds`テーブルを公開スキーマに作成する場合 THEN Row Level Securityを有効化し、`auth.uid() = user_id`のポリシーで保護する

---

### 要件8: UXフロー改善（セットアップ画面の整理）

**目的:** ゴルファーとして、「START」を押してからプレー開始までの手順を直感的にしてほしい。現在の「START → 遷移 → SetupModal」の二段階が分かりにくいため。

#### 受け入れ基準

1. WHEN スタート画面でSTARTボタンが押されたとき THEN アプリはセットアップ（試合名・レート・プッシュ制限回数[0〜5、デフォルト2]・「バーディーでプッシュ復活」オプション[ON/OFF]・チーム編成）をスタート画面上で完結させる（スタートコース設定は不要）
1a. WHEN ゲーム開始時にチーム編成を設定するとき THEN ユーザーはプレイヤーの初期順位（P1〜P4またはP1〜P3+ボギーくん）を決定する（ドラッグ&ドロップまたは選択UI）。チームは毎ホール自動でローテーションされる
1b. WHEN レートを表示・計算するとき THEN アプリは通貨単位（円等）を一切表示せず、数値のみで表示する（賭博法への配慮）
1c. WHEN ラウンド終了後の精算を表示するとき THEN アプリは `Diff × multiplier × レート` の数値のみを表示し、「ポイント」等の中立的な表現を使用する
2. WHEN セットアップが完了したとき THEN アプリはスコア入力画面に直接遷移し、二重のセットアップモーダルを表示しない
3. WHEN スコア入力中に設定変更が必要なとき THEN アプリはヘッダーの設定アイコンから変更できる
4. IF プレイヤー名が初期値（Player A〜D）のまま試合開始しようとした場合 THEN アプリは名前変更を促すが、強制はしない

---

### 要件8b: 動的チームローテーション

**目的:** ゴルファーとして、前ホールの成績に基づいてチームが自動的に組み替えられることで、公平で緊張感のある試合展開を実現したい。

#### 受け入れ基準

1. WHEN ホール1が開始するとき THEN アプリはゲーム開始時にユーザーが設定した初期プレイヤー順（P1〜P4）を使用する
2. WHEN 各ホールが終了するとき THEN アプリは各プレイヤーの個人スコア（昇順・低いほど上位）でP1〜P4を再ランク付けする
3. WHEN 前ホールで同スコアのプレイヤーがいるとき THEN アプリは前ホールのランク順を維持する（タイブレーク）
4. WHEN 各ホールのチームを決定するとき THEN アプリは以下の固定ローテーションパターンをその時点のP1〜P4に適用する:
   - ホール番号 % 3 == 1: P1+P2 vs P3+P4
   - ホール番号 % 3 == 2: P1+P3 vs P2+P4
   - ホール番号 % 3 == 0: P1+P4 vs P2+P3
5. WHEN 3人モードのとき THEN ボギーくんをP4として扱い、毎ホール終了後のスコアランキングにも参加させる（ボギーくんのスコア = そのホールのpar+1）
6. WHEN 3人モードでチームが確定したとき THEN ボギーくんと同チームになったプレイヤーをソロプレイヤーとして扱い、そのホールの獲得/損失ポイントを×2にする
7. WHEN ホール開始前にチーム編成を表示するとき THEN アプリは現ホールのチームとソロプレイヤー（3人時）を明示する

### 要件9: コンポーネント分割とコード品質

**目的:** 開発者として、`ScoreInputScreen.tsx`（現在820行）を保守しやすい構造に分割したい。新機能追加やバグ修正の影響範囲を局所化するため。

#### 受け入れ基準

1. IF `ScoreInputScreen.tsx` のコード行数が300行を超えている場合 THEN 開発者は機能単位でコンポーネントを分割する
2. WHEN スコア入力ロジックを変更する場合 THEN 開発者はUIコンポーネントではなくストアまたはユーティリティを修正する
3. WHEN 同一のダイアログ（履歴など）が複数の画面に重複して存在する場合 THEN 開発者は共通コンポーネントとして抽出する
4. WHERE ビジネスロジックが存在する場合 THE コードは `src/utils/golfLogic.ts` に集約し、UIコンポーネントにロジックを持たせない

---

### 要件10: テストの正確性と網羅性

**目的:** 開発者として、ビジネスロジックのテストが仕様に準拠していることを保証したい。バグを含む実装をテストが「正しい」と承認しないようにするため。

#### 受け入れ基準

1. WHEN `golfLogic.ts` のロジックを変更するとき THEN すべての既存テストが修正後の仕様に対して通過する
2. WHEN バーディーが発生したとき THEN テストはバーディーが倍率ではなくFlipのみに影響することを検証する
3. WHEN キャリーオーバーが連続するとき THEN テストはx2→x4→x6の倍率増加を検証する
4. WHEN プッシュが複数人から宣言されたとき THEN テストは乗算ではなく `2 × totalPushCount` の倍率を検証する
5. WHEN プッシュとキャリーオーバーが同時発生したとき THEN テストは `PushMultiplier + CarryOverMultiplier` の加算倍率を検証する

---

### 要件11: UI日本語化の完全対応

**目的:** ゴルファー（日本人ユーザー）として、アプリ内のすべてのテキストが日本語で表示されてほしい。

#### 受け入れ基準

1. WHEN 言語設定が日本語のとき THEN アプリはすべてのダイアログタイトル・ボタン・エラーメッセージを日本語で表示する
2. IF UIコンポーネントにHardcodedの英語文字列が存在する場合 THEN 開発者はそれをi18nキーに置き換える
3. WHEN 新しいUIテキストを追加するとき THEN 開発者は `ja.ts` と `en.ts` の両方にキーを追加する

---

### 要件12: プッシュ残り回数の正確な表示

**目的:** ゴルファーとして、現在のホールで何回プッシュを使えるか正確に知りたい。使いすぎを防ぐため。

#### 受け入れ基準

1. WHILE スコア入力中 THEN アプリは各プレイヤーの「残りプッシュ回数 / 上限」を表示する
2. WHEN プレイヤーが今ホールでプッシュを選択しているとき THEN アプリはその選択分を残り回数から差し引いた値を表示する
3. WHEN 残りプッシュ回数が0のとき THEN アプリはプッシュボタンをDisabled状態にする
4. WHEN アプリ上のホール9→10に遷移するとき THEN アプリはプッシュ使用回数をリセットする（アプリ内ホール番号は常に1から順番に記録し、ゴルフ場の実際のホール番号と一致させる必要はない）
5. WHEN プッシュ制限が設定されているとき THEN 上限は1プレイヤーあたり・1ハーフあたりの回数で管理し、デフォルトは2回・設定範囲は0〜5回とする
6. WHEN 「バーディーでプッシュ復活」オプションが有効かつプレイヤーがバーディーを取ったとき THEN 次のホールからそのプレイヤーの残りプッシュ回数を+1する（上限キャップなし・複数バーディーで累積可能）
